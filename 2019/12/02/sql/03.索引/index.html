索引

一、什么是索引

​ 索引是对数据库表中一列或多列的值进行排序的一种结构，使用索引可快速访问数据库表中的特定信息。

二、为什么要使用索引

​ 快速查询数据，解释：全表扫描只是针对于数据较少的表中才使用，为了避免全表扫描，我们引入了索引的概念，灵感来源于《新华字典》，对关键信息进行索引存储来进行查询。

https://www.cs.usfca.edu/~galles/visualization/Algorithms.html

三、索引使用的原则（列举）

1.选择唯一性索引：

【选择性】

唯一性索引的值是唯一的，可以更快速的通过该索引来确定某条记录。    select * from table表名  where id = 5;
为常作为查询条件的字段建立索引。    select * from table表名  where    age = 20 and name = "熊大"

2.限制索引的数目：

【浪费空间】 【当前列需要当做条件】

越多的索引，会使更新表变得很浪费时间。

3.删除不再使用或者很少使用的索引

4.最左前缀匹配原则

联合索引的最左前缀匹配原则的成因

    在A B两个字段建立联合索引，where A B   走联合索引，where B A   走联合索引，where A   走联合索引，where B  不走联合索引。

    最左前缀匹配原则，MySQL会一直向右匹配。=和in可以乱序，MySQL的查询优化器会帮你优化成索引可以识别的形式。但是遇到范围查询（> < between like）就停止匹配


联合索引先对第一个字段进行排序，所以一个字段是绝对有序的，然后才会对第二个字段进行排序。

alter table users2 add index userName_password(userName,password);

友情链接：https://blog.csdn.net/qq_16557637/article/details/94324296

5.尽量选择区分度高的列作为索引

【选择性】

四、索引的数据结构（实现方式）

https://www.cnblogs.com/chaosimple/archive/2013/06/12/3132507.html

术语：
根节点:    最顶层的节点
节点:      有子节点的节点
叶子节点：  就是没有子节点的节点
关键字:    搜索的时候判断去左节点和右节点进行搜索的依据
“高度”——从叶节点到目标节点的长度
“深度”——从根节点到目标节点的长度    
“阶数”——节点的最大儿子（分支）个数

1.二叉树

生成索引，建立二叉查找树进行二分查找    

        特性:
        1.所有非叶子结点至多拥有两个儿子（Left和Right）；
        2.所有结点存储一个关键字；
        3.非叶子结点的左指针指向小于其关键字的子树，右指针指向大于其关键字的子树；

2.B-Tree

生成索引，建立B-Tree（平衡多路查找树）结构进行查找。
    特性:
    1.关键字集合分布在整颗树中；
    2.任何一个关键字出现且只出现在一个结点中；
    3.搜索有可能在非叶子结点结束；
    4.其搜索性能等价于在关键字全集内做一次二分查找；
    5.自动层次控制；

3.B+Tree

生成索引，建立B+-Tree结构进行查找。（主流：B+树的查询效率更加稳定、更有利于对数据库的扫描、适合范围查询。）
    特性:
    1.所有关键字都出现在叶子结点的链表中，且链表中的关键字恰好是有序的；
    2.不可能在非叶子结点命中；
    3.非叶子结点相当于是叶子结点的索引，叶子结点相当于是存储（关键字）数据的数据层；
    4.更适合文件索引系统(原因是:所有叶子结点有一个链指针,方便进行)支持范围统计,定位到某个节点之后,可以横向的跨子树进行统计

4.Hash

生成索引，建立Hash结构结构进行查找。（数组+链表,仅仅能满足“=”，“IN”，不能使用范围查询。）  
    缺点:
    1.不能使用范围查询,不考虑
    2.无法用来进行排序运算
    3.大量Hash值相等的情况,效率不一定会比B-Tree效率高

五、为什么B+树更适合做数据库索引【重点】

(最终原因:是为了提升查找效率)

1.查找数据最原始的是【全盘扫描】,但是如果对于几十万上百万数据查询的话,效率就很低了,

2.此时需要进行查找方式的优化,最熟悉的就是【二叉树】

3.但是二叉树每个节点都是一个关键字,要把全部的索引关键字加载到内存,需要很大的内存开销,

如果将一个节点读到内存中的一个盘块的话,B+树/B- 树 使用的盘块越少会进行更少的【IO读写】,

4. 【B- 树和  B+树的多路搜索树】(多路的“多”是相对二叉树中的“二路”的)可以降低树的高度，只需要遍历非子叶节点,提高查找效率

5.B- 树和  B+树之间,B+树所有叶子结点有一个【链指针】,【范围统计】效率更高,所以B+树更适合数据库索引的数据结构

6.【B+树查询效率】比较稳定,每一次查询都是从根节点查询到叶子节点,并且每次的查询的长度相同,每个数据的查询效率相同,稳定的O(logn)

六、MySQL数据库的索引分类

主键索引：它是一种特殊的唯一索引，不允许有空值。一般是在建表的时候同时创建主键索引。

普通索引(Normal)：最基本的MySQL数据库索引，它没有任何限制。

唯一索引(Unique)：它与普通索引类似，不同的就是：MySQL数据库索引列的值必须唯一，但允许有空值。如果是组合索引，则列值的组合必须唯一。

全文索引(Full Text ):  全文索引可以在varchar、char、text类型的列上创建。  ES  搜索引擎

空间索引(SPATIAL):空间索引是对空间数据类型的字段建立的索引,空间索引只能在存储引擎为myISAM的表中创建

七、导致索引失效的情况（列举）

a.模糊查询的某种情况下索引会失效

如果模糊查询%在左边，索引会失效
EXPLAIN SELECT * from student s where  s.last_name like '%铭'

如果模糊查询%在右边，索引会生效
EXPLAIN SELECT * from student s where  s.last_name like '铭%'

b.组合索引什么时候会失效，什么时候会生效

联合索引概念：
    多个字段组合成一个索引，  a ， b ，c

联合索引的最左前缀匹配原则的成因
    在A B两个字段建立联合索引，where A B   走联合索引，where B A   走联合索引，where A   走联合索引，where B  不走联合索引。
最左前缀匹配原则，MySQL会一直向右匹配直到遇到范围查询（> < between like）就停止匹配。=和in可以乱序，MySQL的查询优化器会帮你优化成索引可以识别的形式。

联合索引先对第一个字段进行排序，所以一个字段是绝对有序的，然后才会对第二个字段进行排序。
alter table users2 add index userName_password(userName,password);

联合索引最左边的字段是B+树中的关键字节点。

结论：如果碰到联合索引，查询条件中一定要包含联合索引最左边的字段，否则联合索引不生效。
    前提一定是要精确查询  in 和 and 一定没有问题精确查询
    模糊查询 like  %铭 失效  
    待考证：between

友情链接：https://blog.csdn.net/qq_16557637/article/details/94324296【23 24答案写反了】

c.如果你的sql里面涉及到 d in （a,b,c）不属于模糊查询，属于精确查询

d.索引失效的几种情况 7个索引失效

1、不要在查询中使用 !=、not in、not exists

SELECT * FROM goods WHERE STATUS != 0

可优化为：

SELECT * FROM goods WHERE STATUS = 1 OR STATUS = 2

由于在使用not in时，MySQL会进行全表扫描而不利用索引，因为MySQL认为使用索引反而更麻烦。比如，从 1000 个人中把编号为145、761 这两个学生之外的所有学生资料抄写一遍。你是先找出这两个学生的成绩放一边儿，然后再开始抄写，还是直接开始一份挨一份抄写，看到这两个号直接跳过去？

2、不要在where条件的等号左侧中使用函数

SELECT * FROM goods WHERE YEAR (time) > = '2017'

可优化为：

SELECT * FROM goods WHERE time > = '2017-01-01'

对于带有列的函数计算，MySQL是无法使用索引的（Oracle可以使用函数索引）。在使用函数时，如SUBSTR函数，将'apple'更改为了'app'，此时值发生了改变，导致之前设置的索引无效。

***3、MySQL支持前缀索引，但是不支持后缀索引【普通索引】

作业：把普通索引修改为全文索引，看看走不走索引：

CREATE FULLTEXT INDEX idx_info ON student(last_name) ;

SELECT * FROM goods WHERE NAME LIKE '%A'

可优化为：

SELECT * FROM goods WHERE NAME LIKE 'A%'

全文索引使用:【分词，倒序排序】搜索引擎 ES

使用全文索引
     跟普通索引稍有不同
     使用全文索引的格式：  MATCH (columnName) AGAINST ('string')
     eg:
           SELECT last_name from student  where  MATCH (last_name) AGAINST ('铭')
           当查询多列数据时：
                建议在此多列数据上创建一个联合的全文索引，否则使用不了索引的。

          SELECT * FROM `student` WHERE MATCH(`name`,`address`) AGAINST('聪 广东')

*** 4、设置联合索引时，索引的建立顺序要和where或者order by条件中的字段顺序相同，如建立name、status（分别为key1、key2）的联合索引，此时以name字段为第一查询顺序的索引有效：

SELECT * FROM goods WHERE NAME = 'A' AND STATUS = 0

SELECT * FROM goods WHERE NAME = 'A' 

但以STATUS开头的查询索引都会失效：

SELECT * FROM goods WHERE STATUS = 0 AND NAME='A'

SELECT * FROM goods WHERE STATUS = 0

*** 5、where条件中有or，需将or中的每个字段都加上索引

作业：如果换成and 索引失效情况

SELECT * FROM goods WHERE NAME = 'A' OR STATUS = 0

若仅设置 name 或 status 其中一个字段为索引，索引无效。

6、状态值少的字段不宜使用索引

例如索引列 sex 字段有2个状态值，如果有10w条数据，那么where sex = 0 将访问表中的 5w个数据块，再加上访问索引块，一共要访问大于 5w个的数据块。如果全表扫描，假设10条数据一个数据块，只需访问 1w个数据块，此时全表扫描比利用索引效率更高，因此对于状态值很少的列，不适合建立索引。

八、索引是建立得越多越好吗

数据量小的表不需要建立索引，建立会增加额外的索引开销。                            

数据变更需要维护索引，因此更多的索引意味着更多的维护成本。

更多的索引意味着也需要更多的空间。

索引的选择性

索引的选择性是指索引列中不同值的数目与表中记录数的比。
如果一个表中有2000条记录，表索引列有1980个不同的值，那么这个索引的选择性就是1980/2000=0.99。一个**索引的选择性**越接近于1，这个索引的效率就越高。

九、哪些情况需要创建索引:

总结：主键 ，【条件】 ，最左前缀

主键自动建立唯一索引【主键】

频繁作为查询条件的字段应该创建索引【where查询条件】

查询中与其他表关联的字段,外键关系建立索引【因为外键需要进行表与表之间的where条件查询】 【条件】

单键/组合索引的选择问题,(在高并发下倾向创建组合索引)【最左前缀】

查询中排序的字段,排序字段若通过索引去访问将大大提高排序速度【排序   条件】

查询中统计或者分组字段【分组   条件】

十、那些情况下不要创建索引

总结：数据少，更新多，where条件不使用，选择性不足【性别】

表记录太少【数据少】

经常增删改的表【更新多】

数据重复且分布平均的表字段,因此应该只为经常查询和最经常排序的数据列建立索引. 注意:如果某个数据列包含许多重复的内容,为它建立索引就没有太大的实际效果【不接近选择性】

where条件里用不到的字段不创建索引【where 条件不使用用】

频繁更新的字段不适合创建索引,因为每次更新不单单是更新了记录还会更新索引【更新多】
